如果多个逻辑控制流在时间上重叠执行，称为 **并发(concurrent)**，多核的并发可以实现并行。

并发不仅在内核，也在应用程序：
 - 访问慢速 I/O 设备时，交替执行 I/O 请求和其他有用的工作来利用并发；
 - 和计算机交互的人要求计算机有同时执行多个任务的能力；
 - 通过推迟部分工作以降低延迟（将耗时操作放在后台并发处理）；
 - 在多核机器上进行并行计算；

使用应用级并发的应用程序称为 **并发程序(concurrent program)**，现代操作系统提供了三种基本的构造并发程序的方法：
 - **进程**，每个逻辑控制流都是一个进程，由内核来调度和维护（进程拥有其独立的虚拟地址空间，需要使用显式的进程间通信机制 -- IPC(interprocess communication) 来通信）；
 - **I/O 多路复用**，应用程序在某一进程的上下文中调度逻辑流，一个逻辑流被分为多个阶段，根据当前状态决定下一步，并进行不同逻辑流之间的状态切换，它们共享同一地址空间；
 - **线程**，运行在单一进程上下文中的逻辑流，由内核进行调度，但共享同一进程的虚拟地址空间（进程 与 I/O 多路复用 的混合）；

本章将研究三种不同的并发编程技术，并以 11.4.9 节中的迭代 echo 服务器的并发版本为例。

# 12.1 基于进程的并发编程
构造并发程序最简单的方法就是进程，使用 `fork`、`exec` 和 `waitpid` 之类的函数。

而构造一个并发服务器的方法是：在父进程中接受客户端连接请求，然后创建一个新的子进程来为每个新客户端提供服务，父进程则继续监听新的连接。由于 `fork` 后父进程与子进程共享描述符，所以子进程需要关闭监听描述符，父进程关闭已连接描述符，以避免文件表项泄漏。

通过为每个客户端创建子进程，服务器可以并发地为多个客户端提供服务。

# 12.2 基于 I/O 多路复用的并发编程
由于服务器不仅需要响应网络客户端发起的连接请求，还有用户在键盘上键入。选择了一个就不能响应其他，故采用 **多路复用(I/O multiplexing)** 技术，即使用 `select` 函数，要求内核挂起进程，一次等待多个描述符，谁先准备好就处理谁 -- 将控制返回给应用程序。

`select` 有许多不同的使用场景，在此只讨论第一种场景：等待一组描述符 `fdset` 中某个 `fd` 可以读：
```c
int select(int n, fd_set *fdset, NULL, NULL, NULL);    // 返回已准备好的描述符的非零个数，若出错则为 -1

FD_ZERO(fd_set *fdset);           /* Clear all bits in fdset */
FD_CLR(int fd, fd_set *fdset);    /* Clear bit fd in fdset */
FD_SET(int fd, fd_set *fdset);    /* Turn on bit fd in fdset */
FD_ISSET(int fd, fd_set *fdset);  /* Is bit fd in fdset on? */
// 处理描述符集合的宏。
```
该函数处理类型为 `fd_set` 的集合，也成为 **描述符集合**：
 - 逻辑上是一个大小为 `n` 的位向量：$b_{n-1},…,b_1,b_0$，每个位 $b_k$ 对应一个描述符 $k$；
 - 当 $b_k=1$ 时，表示描述符 $k$ 在集合中；
 - 只可以对描述符集合做这三件事：
	 - 分配它们；
	 - 将一个此种类型的变量赋值给另一个变量；
	 - 用 FD_ZERO、FD_SET、FD_CLR 和 FD_ISSET 宏来修改和检查它们；

`select` 函数有两个输入：称为 **读集合** 的描述符集合(`fdset`) 和该读集合的基数(`n`)。

> `select` 函数会一直阻塞直到读集合中至少有一个描述符可以读了。

同时，`select` 会修改参数 `fdset` 指向的 `fd_set`，指向一个由读集合中准备好可以读的描述组成的一个集合，称为 **准备好集合(ready set)**。由于 `fd_set` 被修改了，为此需要在每次调用 `select` 时读更新读集合。

如图 12-6 展示了可以如何利用 `select` 来实现一个迭代 echo 服务器，可以接受标准输入上的用户命令：
![[NoteAboutStudy/attachments/Pasted image 20260326211405.png]]
 - 检查：`if (FD_ISSET(stdin))` -- 用户输入了命令，调用 `command()` 处理；
 - 检查：`if (FD_ISSET(listenfd))` -- 有客户端连接，调用 `accept、echo、close`；
一个进程可以同时处理多个 I/O，即 I/O 多路复用并发。

虽然使用了 `select` 函数进行阻塞，但仍然不是并发服务器，因为一旦连接到某个客户端就会连续回送输入行，直到客户端关闭连接。因此，如果键入一个命令在服务器和客户端之间结束之前将不会得到响应。为此，更细粒度的多路复用是服务器每次循环至多回送一个文本行。

## 12.2.1 基于 I/O 多路复用的并发事件驱动服务器
I/O 多路复用可以用做并发 **事件驱动(event-driven)** 程序的基础。在事件驱动程序中，等待事件，来了就处理，一般是将逻辑流模型化为状态机。

可以说一个 **状态机(state machine)** 就是一组 **状态**、**输入事件** 和 **转移**，其中每个转移将一个（输入状态和输入事件）对映射到一个输出状态。自循环就是同一输入和输出状态之间的转移。

> 一个状态机从某种初始状态开始执行，每个输入事件都会引发一个从当前状态到下一个状态的转移。

## 12.2.2 I/O 多路复用技术的优劣
### 1. 核心优势（本质：控制力 + 低开销）
#### 1）更强的“程序控制力”
事件驱动模型把调度权从内核拿回到**应用层**
- 进程模型：  
    → 谁先执行，由操作系统调度（不可控）
- 事件驱动：  
    → 自己决定先处理哪个客户端、处理多少
这带来的直接能力是：
- 可以做**优先级调度**
- 可以对不同客户端做**差异化服务**
- 可以实现更复杂的策略（限速、丢弃、延迟处理等）
本质：**调度策略从 “黑盒” 变成 “自己写”**

#### 2）共享数据非常容易
- 只有一个进程
- 所有逻辑流共享同一地址空间
所以：
- 不需要 IPC（管道、共享内存、`socket` 等）
- 不需要复杂同步机制（很多情况下甚至不用锁）
对比：

| 模型       | 数据共享       |
| -------- | ---------- |
| 进程       | 很难（必须 IPC） |
| 线程       | 容易（但要锁）    |
| I/O 多路复用 | 最简单（天然共享）  |

#### 3）性能高（避免上下文切换）
事件驱动模型的关键优势：
- 没有 `fork`（创建进程开销大）
- 没有线程切换
- 没有上下文切换
 所以：
> 在高并发但 I/O 密集场景下，性能通常优于进程模型

#### 4）更容易调试
因为是**单进程顺序执行**
- 可以用 GDB 正常调试
- 不用面对：
    - 多进程
    - 多线程竞态
    - 难复现 bug
这一点在工程中非常关键

### 2. 核心缺点（本质：复杂性 + 局限性）
#### 1）代码复杂度极高（最大痛点）
需要自己维护：
- 状态机
- 事件分发
- 客户端池
- 非阻塞逻辑
对比：

| 模型   | 复杂度 |
| ---- | --- |
| 进程   | 很简单 |
| 线程   | 中等  |
| 事件驱动 | 很复杂 |

并发粒度越细（比如按 “字节” 而不是 “行” 处理），复杂度指数级上升

#### 2）容易被 “慢客户端” 拖累（粒度问题）
如果当前模型的粒度是：“一次读一整行”，而客户端 **只发一半数据就停住**，那么服务器会一直卡在这个逻辑流上。就会导致，其他客户端被阻塞，出现 DoS 风险（慢速攻击）

#### 3）无法利用多核（致命缺陷）
事件驱动服务器：
- 只有一个进程
- 只能用一个 CPU 核心
在现代多核机器上，这是巨大的性能浪费，而像进程和线程都可以多进程。

# 12.3 基于线程的并发编程
**线程(thread)** 就是运行在进程上下文中的逻辑流，虽然本书迄今为止所讲述的程序中的每个进程都是一个线程，但是现代系统也可以编写一个进程里同时运行多个线程的程序。

> 每个线程都由自己的 **线程上下文(thread context)**，包括唯一的整数 **线程 ID(Thread ID, TID)**、栈、栈指针、程序计数器、通用目的寄存器和条件码。所有运行在一个进程里的线程共享该进程的整个虚拟地址空间。

| 机制       | 调度者  | 地址空间 | 通信成本   | 并发控制 |
| -------- | ---- | ---- | ------ | ---- |
| 进程       | 内核   | 不共享  | 高（IPC） | 简单   |
| I/O 多路复用 | 用户程序 | 共享   | 低      | 很复杂  |
| 线程       | 内核   | 共享   | 低      | 中等   |

---
## 12.3.1 线程执行模型
多线程的执行模型在某些方面和多进程的执行模型是相似的，如图 12-12 中所示：
![[NoteAboutStudy/attachments/Pasted image 20260327203530.png]]
即每个进程刚开始它的生命周期时都是单一线程，这个线程称为 **主线程(main thread)**，在某一时刻，主线程创建一个 **对等线程(peer thread)**，从此时开始，两个线程就需要并发地运行 -- 即在时间上交替执行，典型的调度过程是：
1. 主线程运行
2. 因为某些原因让出 CPU：
     - 阻塞（比如 `read`、`sleep`）
     - 时间片用完（被时钟中断）
3. 发生**线程级上下文切换**
4. 对等线程开始执行
5. 再切回主线程
6. 循环往复

### 线程 vs 进程：核心差异
#### 1. 上下文切换成本
线程是共享地址空间，上下文切换的成本比进程切换小很多

#### 2. 组织关系
进程是树结构，而线程是对等关系，所有线程地位平等（主线程的特殊性只有它是第一个运行的线程）。

#### 3. 对等线程的主要影响
 - 一个线程可以杀死任何的另一个线程，也可以等待任意的另一个线程结束（只要是同一进程的线程）；
 - 每个对等线程都能读写相同的共享数据（但每个线程也独有自己的栈和寄存器上下文），就会导致数据竞争；
 - 因为是共享内存的，故需要引入互斥锁、信号量、条件变量之类的来解决同步问题；

## 12.3.2 Posix 线程
**Posix 线程(Pthreads)** 是在 C 程序中处理线程的一个标准接口（在所有的 Linux 系统上都可用）。该接口定义了一些关于程序创建、杀死和回收线程、与对等线程安全地共享数据、通知对等线程系统状态的变化的一些函数。

如图 12-13 所示为一个简单的 Pthreads 程序：
![[NoteAboutStudy/attachments/Pasted image 20260327215859.png]]
 - 主线程创建一个对等线程，然后等待它的终止；
 - 对等线程输出 "Hello, world!\n"，并终止；

主线程负责 “调度”，子线程负责 “干活”。

线程的代码和本地数据被封装在一个 **线程例程(thread routine)** 中，为此声明 `void *thread(void *vargp)`，通过 `void *` 做到输入参数和返回值都可以通过指针传 “任何类型的数据”
 - 如果想传递多个参数给线程例程，可以将参数放到一个结构体中，这样就可以通过传递结构体的指针来传递多参数；

 4. 主线程通过调用 `pthread_join` 函数来阻塞当前进程 -- 主线程，直到 `tid` 对应的对等线程终止（第 8 行）；
 5. 主线程调用 `exit`，终止当时运行在这个进程中的所有线程（第 9 行）；
 6. 第 12 ~ 16 行定义了对等线程的例程，这里只是打印一个字符串，然后就通过执行 `return` 来终止对等线程；

## 12.3.3 创建线程
线程通过调用 `pthread_create` 函数来创建其他线程：
```c
typedef void *(func)(void *);
int pthread_create(pthread_t *tid, pthread_attr_t *attr,
                   func *f, void *arg);
// 若成功返回 0，出错返回非零
```
该函数创建一个新的线程，并带着一个输入变量 `arg`，在新线程的上下文中运行 **线程例程 f**：
 - `pthrea_t *tid` 是新线程的标识符，是一个输出参数；
 - `pthread_attr_t *attr` 能用该参数来改变新创建线程的默认属性（改变这些属性已超出学习的范围，在本书的示例中，总是用一个为 NULL 的参数来调用该函数）；
 - `func *f` 相当于该线程的 `main` 函数（执行该线程就是执行这个）；
 - `void *arg` 是传给线程函数的参数；

当 `pthread_create` 返回时，参数 `tid` 包含新创建线程的 ID。新线程可以通过调用 `pthread_self` 函数来获得自己的线程 ID。
```c
pthread_t pthread_self(void);    // 返回调用者的线程 ID。
```

## 12.3.4 终止线程
一个线程可以通过以下方式来终止：
 - 当线程例程 `return`，线程会 **隐式地** 终止 -- 即线程函数执行完毕，自动结束线程；
 - 通过调用 `pthread_exit` 函数来 **显式地** 终止（可以在任意位置调用） -- 立即终止当前线程（如果是主线程的话，需要等待所有其他的对等线程终止，然后再总之主线程和整个进程，返回值为 `thread_return`）
	```C
	void pthread_exit(void *thread_return);    // 从不返回
	```
 - 某个对等线程调用 Linux 的 `exit` 函数，该函数终止进程以及所有与该进程相关的线程；
 - 其他的对等线程通过以当前线程 ID 作为参数调用 `pthread_cancel` 函数来终止当前线程 -- 一个线程请求取消另一个线程（不是强制终止）
	 ```c
	 int pthread_cancel(pthread_t tid);    // 成功返回 0，出错则为非零
	 ```

## 12.3.5 回收已终止线程的资源
线程通过调用 `pthread_join` 函数等待其他线程终止：
```c
int pthread_join(pthread_t tid, void **thread_return);    // 成功返回 0，出错则为 非零
```
该函数会阻塞直到线程 `tid` 终止，然后讲线程例程返回的通用(`void*`)指针赋值为 `thread_return` 指向的位置，然后回收已终止线程占用的所有内存资源。

|进程|线程|
|---|---|
|`wait()`|`pthread_join()`|
|回收子进程|回收线程|
|僵尸进程|僵尸线程|
但与 `wait` 函数不同，`pthread_join` 函数只能等待一个指定的线程终止，需要逐个管理线程，这使得代码复杂度上升。

## 12.3.6 分离线程
线程在创建开始，可以设置为 **可结合(joinable)** 或者是 **分离的(detached)**，默认情况下是前者：
 - 可结合线程，资源不会自动释放，需要其他线程通过 `pthread_join` 函数来回收和杀死；
 - 分离线程，不能被其他线程回收或杀死，内存资源在线程结束时自动释放；
由于默认情况下，线程被创建成可结合的，为了避免内存泄漏，每个可结合线程要么被其他线程显式地回收，要么通过调用 `pthread_detach` 函数来 “分离“：
```C
int pthread_detach(pthread_t tid);    // 成功返回 0，出错则为非零
```
该函数将线程 `tid` 的回收策略由手动回收变为自动回收（线程可以通过以 `pthread_self()` 为参数的 `pthread_detach` 调用来分离自己）。

## 12.3.7 初始化线程
通过 `pthread_once` 函数来初始化与线程例程相关的状态：
```c
pthread_once_t once_control = PTHREAD_ONCE_INIT;
int pthread_once(pthread_once_t *once_control,
                 void (*init_routine)(void));    // 总是返回 0
```
`once_control` 是一个全局或者静态变量，总是被初始化为 PTHREAD_ONCE_INIT，当第一次用参数 `once_control` 调用 `pthread_once` 时，它将会调用 `init_routine`（没有参数也没有返回值）。而其他线程将会阻塞直到第一个进入的初始化完成。

### 1. 为什么必须 `Malloc`？（竞态条件）
当调用 `pthread_create` 时，如何将已连接描述符传递给对等线程，最明显的方法就是传递一个指向这个描述符的指针：
```C
int connfd;  
connfd = Accept(...);  
pthread_create(..., &connfd);
```
但是是错误写法，根据时间线分析：
```
主线程: connfd = Accept() → 得到 fd1  
主线程: pthread_create(..., &connfd)  
  
子线程: 还没来得及读取 connfd  
  
主线程: connfd = Accept() → 得到 fd2（覆盖）  
  
子线程: 读取 connfd → 读到 fd2 ❌
```
结果：多个线程操作同一个 fd（严重错误）

正确写法（第 20 ~ 21 行）：
```C
connfdp = Malloc(sizeof(int));  
*connfdp = Accept(...);
```
> **使每个线程拥有独立的数据副本**

### 2. 在线程例程中避免内存泄漏
如果不显式地回收线程，就必须分离每个线程，使它们在自己终止时，所占用的内存资源自动回收（第 31 行）。同时也必须小心释放主线程分配的内存块（第 32 行）。

# 12.4 多线程程序中的共享变量
多个线程很容易共享相同的程序变量，但这对于编写程序略为棘手。

为了理解 C 程序中的一个变量是否是共享的，需要解答这些问题：
 1. 线程的基础内存模型是什么？
 2. 根据这个模型，变量实例是如何映射到内存的？
 3. 有多少线程引用这个变量的实例？

> 而变量是否 “共享”，取决于：是否有多个线程引用这个变量的某个实例。

![[NoteAboutStudy/attachments/Pasted image 20260331222232.png]]

## 12.4.1 线程内存模型
一组并发线程运行在一个进程的上下文中：
 - 每个线程都有自己独立的 **线程上下文**，包括线程 ID、栈、栈指针、程序计数器、条件码和通用目的寄存器值；
 - 进程上下文的剩余部分是每个线程和其他线程一起共享的，包括整个用户虚拟地址空间（代码段、`data/bss`区域、堆以及所有的共享库代码和数据区域组成的）以及相同的文件描述符；

> 寄存器永不共享，而虚拟内存总是共享的。
 - 每个线程的执行状态是隔离的；
 - 一个线程的写，对其他线程最终是可见的（不是立刻）；

各自独立的线程的栈被保存在虚拟地址空间的栈区域中，且这些栈通常是被相应的线程独立地访问的。但是由于都是在同一个虚拟地址空间里，并没有硬件隔离，倘若某一线程获得了指向其他线程栈的指针，就可以读写这个栈的任何部分（示例程序在第 26 行展示了这一点，其中对等线程直接通过全局变量 `ptr` 间接引用主线程的栈的内容）。

## 12.4.2 将变量映射到内存
> **变量是否共享，本质取决于：运行时有多少个 “内存实例”**
- 一个实例 → 一定共享
- 多个实例 → 不共享（默认）

多线程的 C 程序中变量根据它们的存储类型被映射到虚拟内存：
 - **全局变量**，是定义在函数之外的变量。在运行时，虚拟内存的 `data/bss` 区域只包含每个全局变量的一个实例，任何线程都可以引用（当一个变量只有一个实例时，只用变量名来表示这个实例）；
 - **本地自动变量**，是定义在函数内部且没有 `static` 属性的变量。在运行时，每个线程的栈都包含自己的所有本地自动变量的实例，**即使多个线程执行同一个线程例程时也是如此**，每个线程都有自己的副本，修改互不影响；
 - **本地静态变量**，是定义在函数内部并有 `static` 属性的变量。虚拟内存的`data/bss` 区域只包含在程序中声明的每个本地静态变量的一个实例（`static` 局部变量 ≈ 全局变量 -- 只是作用域不同）；

|类型|定义位置|实例个数|是否共享|
|---|---|---|---|
|全局变量|函数外|1|✅|
|自动变量|函数内|每线程1个|❌（默认）|
|static变量|函数内|1|✅|

## 12.4.3 共享变量
> 变量 v 是共享的 ⇔ 它的某一个 “运行时实例” 被多个线程引用。

即是否多个线程访问 “同一块内存”（如果有指针 “打破边界”，本地自动变量也可能是共享的）。

# 12.5 用信号量同步线程
共享变量使一些操作变得方便，但同时也引入了 **同步错误(synchronization error)** 的可能性。

> 由于多个线程对同一共享变量进行 “非原子操作”，导致结果不确定。

如图 12-16 中的程序 `badcnt.c`，它创建了两个线程，每个线程都对共享计数变量 `cnt` 加 1：
![[NoteAboutStudy/attachments/Pasted image 20260407222213.png]]
![[NoteAboutStudy/attachments/Pasted image 20260407222225.png]]
因为每个线程都对计数器增加了 `niters` 次，故理论上的结果 `cnt = 2 * niters`，但实际上的结果随机且每次不同。

原因在于 `cnt++` 在底层的实现并不是原子操作，而是：
 1. 从内存读 `cnt`，加载该变量到寄存器；
 2. 在寄存器中执行 +1 的操作；
 3. 将 `cnt` 写回内存；
而线程调度顺序是不可预测的，故每次执行路径可能不同，结果也会不稳定。

## 12.5.1 进度图
**进度图(progress graph)** 将 $n$ 个并发线程的执行模型化为一条 $n$ 维笛卡尔空间中的轨迹线：
 - 每条轴 $k$ 对应于线程 $k$ 的进度；
 - 每个点($I_1, I_2, …, I_n$)代表线程 $k$ 已经完成了指令 $I_k$ 这一状态；
 - 图的原点对应于没有任何线程完成一条指令的 **初始状态**；

如图 12-19 展示了 `badnet.c` 程序第一次循环迭代的二维进度图，点($L_1, S_2$) 对应于线程 1 完成了 $L_1$，线程 2 完成了 $S_2$ 的状态。
![[NoteAboutStudy/attachments/Pasted image 20260407223612.png]]

进度图将指令执行模型化为从一种状态到另一种状态的 **转换**：
 - 转换被表示为一条从一点到相邻点的有向边；
 - 合法的转换是向右或者向上，前者是线程 1 的一条指令完成，后者是线程 2 的一条指令完成；
 - 两条指令不能同一时刻完成 -- 对角线转换是不允许的；
 - 程序也不能反向运行，故向下或向左的转换也是不合法的；

对于某一线程，操作共享变量内容的指令 = **临界区(critical section)**，而每个线程在执行临界区中的指令时，需要对共享变量 **互斥的访问**，称为 **互斥(mutual exclusive)**。

两个线程的临界区 “同时进行” 的区域称为 **不安全区(unsafe region)**。

如图 12-21 展示了变量 `cnt` 的不安全区，是所有可能出错的状态集合：
![[NoteAboutStudy/attachments/Pasted image 20260407224548.png]]
 - 绕开不安全区的轨迹线称为 **安全轨迹线**（每个线程完整执行完临界区）；
 - 进入不安全区的轨迹线称为 **不安全轨迹线**（有线程交叉执行临界区）；
任何安全轨迹线都将正确地更新共享计数器。

为了保证任何共享全局数据结构的并发程序的正确执行，为此需要 **同步线程**，使它们总是有一条安全轨迹线，而一个经典的方法就是基于信号量的思想。

## 12.5.2 信号量

 - P(s)：

	if (s > 0):
	    s--;
	    继续执行;
	else:
	    阻塞（挂起）;
