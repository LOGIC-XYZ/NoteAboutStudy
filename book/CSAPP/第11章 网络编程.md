无论是浏览网页、发送邮件还是在线游戏，所有网络应用都建立在同一套基本编程模型之上。它们具有相似的整体结构，并依赖相同的系统接口（主要是套接字接口）。差别更多体现在业务逻辑，而不是底层机制。

编写网络程序并不是一个孤立主题，它综合运用了操作系统中的关键概念，包括：
- 进程与并发控制
- 信号处理
- 字节序与数据表示
- 内存映射
- 动态内存管理
这些都是构建可靠网络服务的基础能力。

本章将在这些基础上讲述基本的客户端 - 服务器编程模型，以及如何编写使用因特网提供的服务的客户端 -- 服务器程序。

# 11.1 客户端 -- 服务器编程模型
每个网络应用都是基于 **客户端 - 服务器模型** 的。一个应用是由一个 **服务器** 进程和一个或多个 **客户端** 进程组成，服务器管理某种资源，并且通过操作这种资源来为它的客户端进行检索和执行。

客户端 - 服务器模型中的基本操作是 **事务(transaction)**，如图 11-1 所示
![[NoteAboutStudy/attachments/Pasted image 20260302201333.png]]
一个客户端 - 服务器事务由以下四步组成：
 1. 当一个客户端需要服务时，便向服务器发送一个请求 -- 即发起一个事务；
 2. 服务器收到请求后，解释并以适当的方式操作它的资源；
 3. 服务器给客户端发送一个响应，并等待下一个请求；
 4. 客户端收到响应并处理它；

一台主机可以同时运行许多不同的客户端和服务器，一个客户端和服务器的事务可以在同一台或是不同的主机上。

# 11.2 网络
客户端和服务器通常运行在不同的主机上，并且通过计算机网络的硬件和软件资源来通信。在此，仅从程序员的角度讲解一个可行的模型：
 - 对主机而言，网络可以看作一种 I/O 设备，是数据源和数据接收方，如图 11-2 所示
	 - ![[NoteAboutStudy/attachments/Pasted image 20260302212403.png]]
	 - 一个插到 I/O 总线扩展槽的适配器(网络适配器 - 即网卡) 提供了到网络的物理接口，从网络上接受的数据从适配器经过 I/O 和内存总线复制到内存，通常是通过 DMA 传送。同样，数据也能从内存复制到网络；
- 物理上而言，网络是一个按照地理远近所组成的层次系统：
	- 最低层是局域网 LAN。而最流行的局域网是 **以太网(Ethernet)**
		- 一个 **以太网段(Ethernet segment)** 包括一些电缆和一个叫作 **集线器** 的小盒子以及多台主机，通过电缆连接集线器的某一端口和某一主机的适配器，每个网卡有唯一 48 位 MAC 地址，数据以 “帧” 形式发送（帧包含：帧头 + 有效载荷），集线器会将其广播到所有端口，但只有目的主机实际读取它；
		- 使用一些电缆和 **网桥(bridge)**，多个以太网段可以连接成较大的局域网，称为 **桥接以太网(bridge Ethernet)**，这种形式，可以通过学习地址表和只在必要时转发帧更充分地利用电缆带宽，如图 11-4 所示：![[NoteAboutStudy/attachments/Pasted image 20260302214134.png]]
	- 多个不兼容的局域网可以通过 **路由器(router)** 这一特殊的计算机连接起来，组成一个 `internet`（一般意义上的互联网络），每台路由器对于所连接的每个网络都有一个适配器（端口），根据目的地址转发数据，可以连接 LAN 和 WAN(Wide-Area Network，广域网)；

> [!tip]+
>  - 小写 internet：一般意义上的互联网络
> 
>  - 大写 Internet：全球 IP 网络

不同网络采用不同的和不兼容技术的各种局域网和广域网组成，为了能让数据跨越不兼容的网络发送数据，通过一层运行在每台主机和路由器上的协议软件来消除差异。这种协议控制主机和路由器该如何工作来实现数据传输，为此必须提供两种基本能力：
 - **命名机制**，统一地址格式，每台主机都会分配 internet address（IP 地址），这个地址唯一地标识了该台主机；
 - **传送机制**，统一数据打包方式，把数据封装成 “包”（packet），包 = 包头 + 有效载荷，而包头包含源地址和目的地址；

如图 11-7 展示了主机和路由器如何使用互联网络协议在不兼容的局域网间传送数据的一个实例：
![[NoteAboutStudy/attachments/Pasted image 20260302215351.png]]
由两个局域网通过一台路由器连接而成，一个客户端运行在主机 A 上，主机 A 与 LAN1 相连，它发送一串数据字节到运行在主机 B 上的服务器端，主机 B 则连接在 LAN2 上：
1. 运行在主机 A 上的客户端进行一个系统调用，从客户端的虚拟地址空间复制数据到内核缓冲区中。 
2. 主机 A 上的协议软件通过在数据前附加互联网络包头和 LAN1 帧头，创建了一个 LAN1 的帧。互联网络包头寻址到互联网络主机 B。LAN1 帧头寻址到路由器。然后它传送此帧到适配器。注意，LAN1 帧的有效载荷是一个互联网络包，而互联网络包的有效载荷是实际的用户数据。这种封装是基本的网络互联方法之一。
3. LAN1 适配器复制该帧到网络上。 
4. 当此帧到达路由器时，路由器的 LAN1 适配器从电缆上读取它，并把它传送到协议软件。 
5. 路由器从互联网络包头中提取出目的互联网络地址，并用它作为路由表的索引，确定向哪里转发这个包，在本例中是 LAN2。路由器剥落旧的 LAN1 的帧头，加上寻址到主机 B 的新的 LAN2 帧头，并把得到的帧传送到适配器。 
6. 路由器的 LAN2 适配器复制该帧到网络上。 
7. 当此帧到达主机 B 时，它的适配器从电缆上读到此帧，并将它传送到协议软件。 
8. 最后，主机 B 上的协议软件剥落包头和帧头。当服务器进行一个读取这些数据的系统调用时，协议软件最终将得到的数据复制到服务器的虚拟地址空间。

# 11.3 全球 IP 因特网
如图 11-8 展示了一个因特网客户端 - 服务端应用程序的基本硬件和软件组织：
![[NoteAboutStudy/attachments/Pasted image 20260304212315.png]]
每台因特网主机都运行实现 **TCP/IP 协议** 的软件，几乎每个现代计算机系统都支持这个协议。因特网的客户端和服务器混合使用 **套接字接口** 函数和 Unix I/O 函数来进行通信。通常将套接字函数实现为系统调用，由内核态的 TCP/IP 协议实现。

TCP/IP 并非单一协议，而是一组分层协议：
 - IP：提供基本的主机命名（IP 地址）和数据报递送机制，但不保证可靠性。丢包、重复包不负责处理。
 - UDP：在 IP 之上提供进程间通信，但仍不保证可靠性。
 - TCP：在 IP 之上提供可靠、全双工、面向连接的进程间通信。保证数据按序、无重复、无丢失地传输。
（在本章仅从应用编程角度出发，关注 TCP 和 IP 提供的抽象能力）

从应用开发者角度，可以把因特网看作满足以下三个性质的系统：
 1. 全球主机被抽象为一组 32 位 IP 地址（IPv4）。  
 2. IP 地址进一步映射为可读的因特网域名。  
 3. 不同主机上的进程可以通过连接进行通信。

> [!tip]+
> IPv4 使用 32 位地址。IPv6 使用 128 位地址，设计目标是替代 IPv4。
> 虽然 IPv6 提出较早，但长期以来 IPv4 仍是主流。IPv4 使用 32 位地址。IPv6 使用 128 位地址，设计目标是替代 IPv4。

## 11.3.1 IP 地址
一个 IP 地址就是一个 32 位无符号整数。网络程序将 IP 地址存放在如下所示的 IP 地址结构中：
```c
struct in_addr{
	uint32_t s_addr;
};
```
把一个标量地址存放在结构中，是一个历史遗留问题，本来一个 `uint32_t` 就可以了，但是已有大量应用，更改太迟了。

因为不同机器的字节序可能不同，小端或者大端，为了保证跨机器一致性，TCP/IP 为任意整数数据项定义了统一的 **网络字节顺序(network byte order)** -- 大端字节顺序。

为此，Unix 提供了下面这样的函数来在网络和主机字节顺序间实现转换：
```c
uint32_t htonl(uint32_t hostlong); 
uint16_t htons(uint16_t hostshort); 
// 返回：按照网络字节顺序的值

uint32_t ntohl(uint32_t netlong); 
uint16_t ntohs(unit16_t netshort); 
// 返回：按照主机字节顺序的值。
```
即凡是要放进网络结构体里的整数，先 hton*，凡是从网络结构体读出来的整数，先 ntoh*（l 是 32 位，s 是 16 位）。

IP 地址通常是以一种称为**点分十进制表示法**来表示的，这里，每个字节由它的十进制值表示，并且用句点和其他字节间分开。

在 Linux 系统上，能够使用 HOSTNAME 命令来确定自己主机的点分十进制地址。应用程序使用 `inet_pton` 和 `inet_ntop` 函数来实现 IP 地址和点分十进制之间的转换。
```c
#include <arpa/inet.h>

int inet_pton(AF_INET, const char *src, void *dst);
// 返回：若成功则为 1，若 src 为非法点分十进制地址则为 0，若出错则为 -1。

const char *inet_ntop(AF_INET, const void *src, char *dst, socklen_t size);
// 返回：若成功则指向点分十进制字符串的指针，若出错则为 NULL。
```

## 11.3.2 因特网域名
因特网客户端和服务器互相通信所使用的是 IP 地址，但是为了更人性化，因特网也定义了 **域名(domain name)**，以及一种将域名映射到 IP 地址的机制。域名就是一串用句点分割的单词（whaleshark.ics.cs.cmu.edu）。

域名集合形成了一个层次结构，每个域名编码了在这个层次中的 “位置”，如图 11-10 展示了域名层次结构的一部分：
![[NoteAboutStudy/attachments/Pasted image 20260304224510.png]]
树的节点表示域名，反向到根的路径形成了域名。子树称为**子域**（subdomain）。
 - 层次结构中的第一层是一个未命名的根节点；
 - 下一层是一组**一级域名**（first-level domain name），由非营利组织 ICANN（Internet Corporation for Assigned Namesand Numbers，因特网分配名字数字协会）定义。常见的第一层域名包括 com、edu、gov、org 和 net；
 - 下一层是**二级**（second-level）域名，例如 **cmu.edu**，这些域名是由 ICANN 的各个授权代理按照先到先服务的基础分配的。一旦一个组织得到了一个二级域名，那么它就可以在这个子域中创建任何新的域名了，例如 **cs.cmu.edu**；

网络通信时，客户端和服务器真正使用的是 IP 地址，而不是域名。因此必须有一个机制把 “域名 → IP 地址” 转换，这个系统就是 DNS（Domain Name System）。DNS 是一个**全球分布式数据库系统**。它存储的是大量的 “主机条目结构(host entry structure)”，每条记录定义：一组域名 ↔ 一组 IP 地址。

有多种典型的映射关系：
 - 一对一映射，一个域名对应一个 IP 地址；
 - 多对一映射，多个域名对应同一个 IP 地址（通常是别名机制或逻辑别名）；
 - 多对多映射（最常见），一个域名对应多个 IP 地址（这是**负载均衡**和**高可用性设计**的基础。如果一台服务器宕机，客户端可以尝试其他 IP）；
 - 合法但不可解析的域名，是命名空间中的节点，但它们本身没有对应的主机 IP（不是每个域名都代表一台主机）；

可使用 Linux 的 NSLOOKUP 程序来探究 DNS 映射的一些属性，这个程序能展示与某个 IP 地址对应的域名。

同时，每台因特网主机都有本地定义的域名 `localhost`，这个域名总是映射为 **回送地址(loopback address)** -- 127.0.0.1，不会经过网络接口，而是直接回到本机协议栈。这对调试客户端 - 服务器程序非常关键。

## 11.3.3 因特网连接
在因特网上，客户端和服务器通过 “连接” 进行通信，并具有三个关键特征：
 - **点对点**，一次连接只发生在两个进程之间，不是广播，不是多播；
 - **全双工**，双方可以同时发送和接收数据，不是单向的；
 - **可靠**，只要底层网络没有灾难性故障（例如链路物理断裂），发送方发出的字节流会按顺序、无丢失地到达接收方（这是 TCP 协议提供的语义保障，只是传输语义上的 “可靠”，连接可能断开，但正常工作期间，顺序、完整性是可靠的）；

连接的每一端都称为一个 **套接字(socket)**。套接字并不是抽象概念，而是在操作系统中以文件描述符形式存在的通信端点。每个套接字都有一个“套接字地址”，由两部分构成：**IP 地址 + 16 位端口号**，形式写成：`地址:端口`
 - 当客户端发起连接时：客户端端口由内核自动分配，这种端口叫 **临时端口(ephemeral port)**，通常范围在高位端口区间（例如 49152–65535）；
 - 服务器则不同：服务器通常监听一个固定端口，这个端口称为 **知名端口(well-known port)**，它与某个特定服务绑定，例如：HTTP → 80、SMTP → 25
在 Linux 系统中，服务名与端口号的映射定义在：`/etc/services` 文件中。

一个连接是由它两端的套接字地址唯一确定的，这对套接字地址称为 **套接字对(socket pair)**，由这个四元组表示：`(cliaddr:cliport, servaddr:servport)`，前者是客户端的套接字地址，后者是服务器端的套接字地址。如图 11-11 展示了一个 Web 客户端和一个 Web 服务器之间的连接：

# 11.4 套接字接口
**套接字接口(socket interface)** 是一组系统调用函数，用来构建网络应用程序。程序通过这些函数创建连接、发送数据、接收数据，从而实现不同主机进程之间的通信。

在实际程序中，套接字接口函数通常与 **Unix I/O 函数（read、write 等）结合使用**。其基本思想是：先通过套接字函数建立通信通道然后像读写文件一样通过 I/O 函数读写数据。

套接字接口已经成为事实上的标准接口，几乎所有现代操作系统都实现了它，例如：Unix / Linux、Windows、macOS，因此基于 socket 的网络程序通常具有很好的可移植性。

## 11.4.1 套接字地址结构
从 **Linux 内核视角**看：
- 套接字是 **网络通信的一个端点**；
- 两个端点连接起来就形成一条通信连接；
从 **用户程序视角**看：
- 套接字就是一个 **文件描述符**；
- 可以像文件一样用 `read` / `write` 操作；

因特网的套接字地址存放在如图 11-13 所示的类型为 `sockaddr_in` 的 16 字节结构中：
![[NoteAboutStudy/attachments/Pasted image 20260306205747.png]]
 - `sin_family` 成员是表示所使用的通信协议（AF_INET 是 IPv4、AF_INET6 是 IPv6、AF_UNIX 是 Unix 本地通信）；
 - `sin_port` 成员是一个 16 位的端口号（大端）；
 - `sin_addr` 成员是一个 32 位的 IP 地址（大端）；
 - `sin_zero` 是填充字段，为了和通用结构 `sockaddr` 对齐；
而图中结构 `sockaddr` 则是通用结构，使网络接口支持多种通信协议。

且系统调用的一些函数的参数类型是 `sockaddr`，为此，需要强制类型转换：
```c
struct sockaddr_in addr;  
bind(sockfd, (struct sockaddr *)&addr, sizeof(addr));
```
也就是：`sockaddr_in  →  sockaddr`，这样 API 才能支持 **不同协议地址结构**。

> [!tip]- _in 后缀
> `sockaddr_in` 的 `_in` 不是 `input`，而是：`internet`，即 IPv4 网络地址结构。
> 
> 类似的还有：
>  - sockaddr_in	IPv4
>  - sockaddr_in6	IPv6
>  - sockaddr_un	Unix socket

## 11.4.2 socket 函数
客户端和服务器使用 `socket` 函数来创建一个 **套接字描述符(socket descriptor)**。
```C
int socket(int domain, int type, int protocol);    // 若成功返回非负描述符，若出错则为 -1
```
 - `domain` 表示使用什么网络协议；
 - `type` 表示通信类型，比如 SOCK_STREAM（TCP）、SOCK_DGRAM（UDP）、SOCK_RAW（原始套接字）；
 - `protocol` 表示具体协议，一般写 `0`，即让系统根据 `domain + type` 自动选择协议；

不过，最好的方法是用 `getaddrinfo` 函数 自动生成参数（11.4.7 节）。

> socket 返回的 clientfd 描述符仅是部分打开的，还不能用于读写，因为连接还没建立。而完成打开套接字的工作取决于是客户端还是服务器。

## 11.4.3 connect 函数
客户端通过调用 `connect` 函数来建立和服务器的连接。
```C
int connect(int clientfd, const struct sockaddr *addr, socklen_t addrlen);    // 成功返回 0，出错则为 -1
```
 - `clientld` 即上一节 `socket` 函数所返回的描述符，是要建立连接的 `socket`；
 - `addr` 是服务器地址（通过 `struct sockaddr *` 来让函数支持多种协议）；
 - `addrlen` 是传进来的地址结构体的大小；

当调用这个函数时，内核会做几件事：
 1. 根据 `addr` 找到服务器 IP + 端口
 2. 给客户端自动分配一个 **临时端口**
 3. 发送 TCP 三次握手
 4. 等待服务器响应
 5. 如果握手成功，建立 TCP 连接
所以 `connect` **是阻塞的**：它会一直等到，连接成功或发生错误，才返回。

> 对于 `socket`，最好的方法是用 `getaddrinfo` 来为 `connect` 提供参数（见 11.4.8 节）。

## 11.4.4 bind 函数
通过 `bind` 函数来把 addr 指定的地址绑定到 sockfd 这个 socket 上：
```c
int bind(int sockfd, const struct sockaddr *addr, socklen_t addrlen);    // 成功返回 0，出错 -1
```
 - `sockfd` 是 `socket` 函数返回的 socket 描述符；
 - `addr` 是服务器地址结构体；
 - `addrlen` 是地址结构体的大小；

客户端通常不需要手动调用 `bind` 函数，原因是：**内核会自动分配临时端口**。

bind 的作用是把服务器 socket 和一个具体的 IP 地址 + 端口号绑定起来，使得客户端能够通过这个地址连接到服务器。

## 11.4.5 listen 函数
客户端是发起连接请求的主动实体，而服务器是等待来自客户端的连接请求的被动实体。

默认情况下，内核会认为 `socket` 函数创建的描述符对应于 **主动套接字(active socket)**，它存在于一个连接的客户端。服务器通过调用 `listen` 函数告诉内核，描述符是被服务器而不是客户端使用的。
```c
int listen(int sockfd, int backlog);    // 成功则返回 0，出错则为 -1
```
把普通 socket 转换为监听 socket（listening socket），用以等待客户端的连接请求。

当多个客户端同时连接服务器时，连接请求不会立即被处理，而是先进入**等待队列**。如果服务器还没来得及调用 `accept()`，这些请求就会被放进一个**连接队列**。而 `backlog` 的作用就是：**指定这个队列最多可以排多少个等待连接**。不过，这个参数的确切含义需要 TCP/IP 知识，超出了讨论的范围，通常会把它设置为一个较大的值。

## 11.4.6 accept 函数
服务器通过调用 `accept` 函数来等待来自客户端的连接请求。
```c
int accept(int listenfd, struct sockaddr *addr, int *addrlen);    // 成功返回非负连接描述符，出错返回 -1
```
该函数等待来自客户端的连接请求到达监听描述符 `listenfd`，然后在 `addr` 中填写客户端的套接字地址，并返回一个 **已连接描述符(connected descriptor)**，用于和客户端通信。

监听描述符相当于服务器的前台，专门用于等待连接、记录连接，通常被创建一次并存在于服务器的整个生命周期；而已连接描述符，服务器每次接受连接请求时都会创建一个，只存在于服务器为该请求的客户端服务的过程中，用于传送数据（读/写）。

如图 11-14 描绘了监听描述符和已连接描述符的角色，是 TCP 建立连接后的行为：
![[NoteAboutStudy/attachments/Pasted image 20260307213352.png]]
1. 服务器：`connfd = accept(listenfd,...)`，此时，服务器阻塞等待客户端连接；
2. 客户端：`connect(clientfd,...)`，客户端发出：TCP 连接请求到服务器的 `listenfd`；
3. 内核完成连接建立：clientfd  <——TCP连接——>  connfd，此时：`accept 返回 connfd`、`connect 返回`，双方都可以通信。通信方式：
	- 客户端：`write(clientfd)`、`read(clientfd)`
	- 服务器：`write(connfd)`、`read(connfd)`

## 11.4.7 主机和服务的转换
Linux 提供了一些强大的函数（称为 `getaddrinfo` 和 `getnameinfo`）实现二进制套接字地址结构和主机名、主机地址、服务名和端口号的字符串表示之间的相互转化。当和套接字接口一起使用时，这些函数能使程序员编写独立于任何特定版本的 IP 协议的网络程序。

---
### 1. getaddrinfo 函数
`getaddrinfo` 函数将主机名、主机地址、服务名和端口号的字符串表示转化成套接字地址结构（这个函数是可重入的 -- 见 12.7.2 节，适用于任何协议）：
```C
int getaddrinfo(const char *host, 
			    const char *service, 
			    const struct addrinfo *hints, 
			    struct addrinfo **result);        // 成功返回 0，出错为非零的错误代码

void freeaddrinfo(struct addrinfo, *result);      // 无返回值；
const char *gai_strerro(int errcode);             // 返回错误消息
```
 - `host` 可以是域名也可以是 IP 地址亦或是 NULL（不转换）；
 - `service` 可以是服务名（如 `http`），也可以是十进制端口号（如 `80`）；
 - `hints` 控制返回结果的过滤条件，是可选参数，如图 11-16 所示为其结构：![[NoteAboutStudy/attachments/Pasted image 20260307215629.png]] 只能设置 `ai_flag`、`ai_family`、`ai_socktype`、`ai_protocol` 这几个字段，其他字段需要设置为 0（或 NULL），在实际使用中，可用 `memset` 将整个结构清零，然后有选择地设置一些字段
	 - `ai_flag` 字段是一个位掩码，可以进一步修改默认行为，可以通过把各种值用 OR 组合起来得到该掩码，以下是一些有用的值：
		 - AI_ADDRCONFIG。如果在使用连接，推荐使用这个标志，**只返回本机支持的协议**（本机是 IPv4 就返回 IPv4）；
		 - AI_CANONNAME。`ai_canonname` 字段默认为 NULL。如果设置了该标志，则将函数返回的链表中的第一个 `addrinfo` 结构的 `ai_canonname` 字段指向 `host` 的标准域名（如 `www.google.com`，返回 `google.com`）；
		 - AI_NUMERICSERV。参数 `service` 可以是服务名或端口号，设置这个标志即是强制 `service` 是端口号；
		 - AI_PASSIVE。服务器设置该标志即是表明接受所有 IP 地址的请求，此时 `host` 为 NULL，得到的套接字地址结构中的地址字段会是 **通配符地址(wildcard address，`0.0.0.0:8080`)**；
	 - `ai_family` 字段表明返回的是 IPv4 还是 IPv6，前者设置为 AF_INET，后者设置为 AF_INET6；
	 - `ai_socktype` 字段表明 `socket` 类型（TCP、UDP、原始套接字），通过设置 SOCK_STREAM 可只返回 TCP 类型；
	 - 当 `getaddrinfo` 创建输出列表中的 `addrinfo` 结构时，会填写每个字段，除了 `ai_flags`。`ai_addr` 字段指向一个套接字地址结构，`ai_addrlen` 字段给出这个套接字地址结构的大小，而 `ai_next` 字段指向列表中下一个 `addrinfo` 结构。其他字段描述这个套接字地址的各种属性；
 - `result` 则返回一个指向 `addrinfo` 结构的链表，其中每个结构指向一个对应于 `host` 和 `service` 的套接字地址结构，如图 11-15 所示：![[NoteAboutStudy/attachments/Pasted image 20260307220108.png]]，每个节点都是一个可能的地址
	 - 在客户端调用了这个函数之后，会遍历这个列表，依次尝试每个套接字地址，直到调用 `socket` 和 `connect` 成功建立起了连接。类似地，服务器会尝试遍历直到 `socket` 和 `bind` 成功；
	 - 为避免内存泄漏，应用程序必须在最后调用 `freeaddrinfo`，释放该链表；
	 - 如果 `getaddrinfo` 返回非零的错误代码，应用程序可以调用 `gai_streeror`，将该代码转换成消息字符串；

### 2. getnameinfo 函数
与前者相反，`getnameinfo` 函数则是将一个套接字地址结构转换成相应的主机和服务名字字符串，且也是可重入的、与协议无关的。
```c
int getnameinfo(const struct sockaddr *sa,  
				socklen_t salen,  
				char *host,  
				size_t hostlen,  
				char *service,  
				size_t servlen,  
				int flags);
```
> `getnameinfo` 函数将套接字地址结构 `sa` 转换成对应的主机和服务名字符串，并将它们复制到 `host` 和 `serveice` 缓冲区。如果 `getnameinfo` 返回非零的错误代码，应用程序可以调用 `gai_strerror` 把它转化成字符串（如果不想要主机名，可以把 `host` 设置为 NULL，`hostlen` 设置为 0，对于服务字段也是如此 -- 不过两者必须设置其中之一）。
 - `sa` 指向大小为 `salen` 字节的套接字地址结构；
 - `host` 指向大小为 `hostlen` 字节的缓冲区；
 - `service` 指向大小为 `servlen` 字节的缓冲区；
 - `flags` 则是一个位掩码用于修改默认行为，以下是两个有用的值：
	 - NI_NUMERICHOST。默认返回 `host` 中的域名，通过设置该标志可使该函数返回一个数字地址字符串；
	 - NI_NUMERICSERV。默认会检查 `/etc/services`，可能会返回服务名而不是端口号，通过设置该标志会使该函数跳过查找，返回端口号；

如图 11-17 展示了一个简单的程序，称为 HOSTINFO，它使用这两个函数展示出域名和它相关联的 IP 地址之间的映射（类似 11.3.2 节中的 NSLOOKUP 程序）：

调用 `getaddrinfo` 之后，会遍历 `addrinfo` 结构，用 `getnameinfo` 将每个套接字地址转换成点分十进制地址字符串。遍历完列表之后，调用 `freeaddrinfo` 释放这个列表。

# 11.5 Web 服务器
在本节中，将讲述如何利用网络编程的基本概念来创建一个麻雀虽小五脏俱全的 Web 服务器。

---
## 11.5.1 Web 基础
Web 客户端和服务器之间的交互用的是一个基于文本的应用级协议，叫作 **HTTP(Hypertext Transfer Protocol， 超文本传输协议)**。
 - 一个 Web 客户端（即浏览器）打开一个服务器的因特网连接，然后请求某些内容；
 - 服务器响应所请求的内容，然后关闭连接；
 - 浏览器读取这些内容，并显示在屏幕上；

Web 服务和常规的文件检索服务(例如 FTP) 的区别：后者只是传文件，前者除了传文件还包括显示规则 -- 称为 **HTML(Hypertext Markup Language，超文本标记语言)** 的语言（是标记语言不是编程语言）来编写。

一个 HTML 程序(页) 包含指令(标记)，告诉浏览器如何相似这页中的各种文本和图形对象，例如：
```html
<b> Make me bold! </b>
```
告诉浏览器用粗体字类型输出 `<b>` 和 `</b>` 标记之间的文本。

HTML 最强大之处是一个页面可以包含指针(**超链接**)，例如：
```html
<a href="http://www.cmu.edu/index.html">  
Carnegie Mellon  
</a>
```
告诉浏览器高亮显示文本对象 “Carnegie Mellon”，并且创建一个超链接，它指向存放在 CMU Web 服务器上叫做 `index.html` 的 HTML 文件。如果用户单击了这个高亮文本对象，浏览器就会从 CMU 服务器中请求相应的 HTML 文件并显示它。

## 11.5.2 Web 内容
对于 Web 客户端和服务器而言，**内容** 是与一个 **MIME(Multipurpose Internet Mail Extensions，多用途的网际邮件扩充协议)** 类型相关的字节序列。如图 11-23 所示为一些常用的 MIME 类型：
![[NoteAboutStudy/attachments/Pasted image 20260314201555.png]]
根据类型让浏览器决定如何显式。

Web 服务器以两种不同的方式向客户端提供内容：
 - **静态内容(static content)** -- 取一个磁盘文件，并将它的内容返回给客户端，而返回文件给客户端的过程称为 **服务静态内容(serving static content)**；
 - **动态内容(dynamic content)** -- 将运行一个可执行文件的所产生的输出返回给客户端，而运行程序并返回它的输出到客户端的过程称为 **服务动态内容(serving dynamic content)**；

每条由 Web 服务器返回的内容都是和它所管理的某个文件相关联的，这些文件中的每一个都有一个唯一的名字，称为 **URL(Universal Resource Locator，通用资源定位符)**。

例如：`http://www.google.com:80/index.html`；
结构：`protocol://host:port/path`；
 - 客户端使用 `protpcol://host:port` ，决定使用的协议、所要连接的主机和端口号；
 - 服务器使用 `/index.html`，决定读所读的文件；

例如：`http://bluefish.ics.cs.cmu.edu:8000/cgi-bin/adder?15000&213`；
 - `/cgi-bin/adder` 是程序；
 - `15000&213` 是参数（`?` 之后是参数，`&` 用以分隔参数）；

> 确定一个 URL 指向的是静态内容还是动态内容没有标准的规则。每个服务器对它所管理的文件都有自己的规则。（可以确定一组目录，这个里面放所有的可执行文件或者音频文件、视频文件之类的）

> URL 路径不是根目录。而是被请求内容类型的主目录
```
/index.html
→ /usr/httpd/html/index.html
```

> `/` 表示默认主页 -- `/index.html` 之类的

## 11.5.3 HTTP 事务
HTTP 事务本质是：
 - 客户端发送文本请求；
 - 服务器返回文本响应 + 数据；
且 HTTP 是基于 TCP 的文本协议。

可以使用 Linux 的 TELNET 程序来和因特网上的任何 Web 服务器执行事务，如图 11-24 所示为使用 TELNET 向 AOL Web 服务器请求主页：
![[NoteAboutStudy/attachments/Pasted image 20260315170541.png]]
在第 1 行，从 Linux shell 运行 TELNET，要求它打开一个到 AOL Web 服务器的连接。TELNET 向终端打印三行输出，打开连接，然后等待输入文本（第 5 行）。每次输入一个文本行，并键入回车键，TELNET 会读取该行，在后面加上回车和换行符号（在 C 的表示中为 `\r\n`），并且将这一行发送到服务器（**HTTP 标准要求每个文本行都由一对回车和换行符来结束**）。为了发起事务，输入一个 HTTP 请求（第 5 ~ 7 行）。服务器返回 HTTP 响应（第 8 ~ 17 行），然后关闭连接（第 18 行）。

---
### 1. HTTP 请求
一个 **HTTP 请求** 的组成是：一个 **请求行(request line)**（第 5 行），后面跟随零个或多个 **请求报头(request header)** （第 6 行），再跟随一个空的文本行来终止报头列表（第 7 行）
 - 请求行 -- `method URI version`：
	 - `method`，包括 GET、POST、OPTIONS、HEAD、PUT、DELETE 和 TRACE
		 - 本书只讨论广为应用的 GET 方法，大多数 HTTP 请求都是这种类型。该方法指导服务器生成和返回 **URI(Uniform Resource Identifier，统一资源标识符)** 标识的内容（URI 是相应的 URL 的后缀，包括文件名和可选的参数 -- 给服务器的那部分）；
	 - `version`，表明该请求遵循的 HTTP 版本，最新的版本是 `HTTP/1.1`，`HTTP/1.0` 是从 1996 年沿用至今的老版本：
		- `HTTP/1.0`，一次连接一次请求；
		- `HTTP/1.1`，定义了一些附加的报头，为诸如缓冲和安全等高级特性提供支持，同时还允许客户端和服务器在同一条 **持久连接(persistent connection)** 上执行多个事务；
		- 在实际使用中，两个版本是互相兼容的，老久的版本会忽略新的版本中的报头；
 - 请求报头 -- `name: value`：
	 - 在 HTTP/1.1 要求中必须有 `Host`，HTTP/1.0 是不需要的，因为一个 IP 可以有多个网站，服务器需要知道域名（**代理缓存(proxy cache)** 会使用 Host 报头，这个代理缓存有时作为浏览器和管理被请求文件的 **原始服务器(origin server)** 的中介。客户端和原始服务器之间，可以有多个代理，即所谓的 **代理链(proxy chain)**。Host 报头中的数据指示了原始服务器的域名，使得代理链中的代理能够判断它是否可以在本地缓存中拥有一个被请求内容的副本）；
 - 空文本行终止报头，并指示服务器发送被请求的 HTML 文件；

### 2. HTTP 响应
一个 **HTTP 响应** 的组成是：一个 **响应行(response line)**（第 8 行），后面跟随着零个或多个 **响应报头(response header)**（第 9 ~ 13 行），再跟随一个终止报头的空行（第 14 行），再跟随一个 **响应主体(response body)**（第 15 ~ 17 行）
 - 响应行 -- `version status-code status-message`：
	 - `version` 字段描述的是响应所遵循的 HTTP 版本；
	 - `status-code` 状态码是一个 3 位的正整数，指明对请求的处理（给服务器用的）；
	 - `status message` 状态消息给出与状态码等价的描述文本（用以给人类看的）；
	 - 如图 11-25 列出了一些常见的状态码，以及它们相应的消息 ![[NoteAboutStudy/attachments/Pasted image 20260315173800.png]]
 - 响应报头提供了关于响应的附加信息：
	 - `Content-Type`（第 12 行），告诉客户端响应主体中内容的 MIME 类型；
	 - `Content-Length`（第 13 行），用来指示响应主体的字节大小；
- 响应主体包含着被请求的内容；

## 11.5.4 服务动态内容
对于动态内容，需要运行程序，并将输出返回给客户端。那么
 - 客户端如何将程序参数传递给服务器；
 - 服务器如何将参数以及信息传递给所创建的子进程；
 - 子进程将输出发送到何处；
 为此，通过一个称为 **CGI(Common Gateway Interface，通用网关接口)** 的标准来解决这些。
---
### 1. 客户端如何将程序参数传递给服务器
GET 请求的参数在 URI 中传递，`?` 分隔了文件名和参数，而每个参数都用 `&` 字符分隔开。且参数中不允许有空格，需要使用 `%20` 来表示空格。

POST 请求的参数在请求主体中。

### 2. 服务器如何将参数传递给子进程
当服务器接收一个关于某个程序的请求后，调用 `fork` 来创建一个子进程，并调用 `execve` 在子进程的上下文中执行 “某个程序”。

像这样的程序，常常被称为 CGI 程序（因为许多 CGI 程序是用 `Perl` 脚本编写的，所以 CGI 程序也常被称为 CGI 脚本）。

在调用 `execve` 之前，子进程将 CGI 环境变量 QUERY_STRING 设置为参数，在程序运行时可用 Linux getenv 函数来引用。
